# 接入本地模型的方法论

## 1. 核心目标

接入本地模型不是简单执行一次模型推理，而是建立一条稳定、可验证、可替换的能力链路：

```text
业务输入
  -> 输入校验与预处理
  -> 模型适配层
  -> 本地推理运行时
  -> 结构化结果
  -> 业务接口与界面
```

工程上的重点是隔离业务逻辑与模型实现。业务层只依赖统一的输入输出契约，不直接依赖某个模型库的对象、异常和返回格式。

## 2. 开始前先明确约束

接入模型前应明确以下问题：

| 问题 | 示例 |
| --- | --- |
| 模型完成什么任务 | 音频转写、文本生成、图像分类 |
| 输入是什么 | 文件路径、字节流、文本、张量 |
| 输出是什么 | 分段字幕、标签、结构化 JSON |
| 推理设备 | CPU、CUDA GPU、Apple Silicon |
| 延迟目标 | 离线批处理、近实时、实时 |
| 数据约束 | 是否允许上传云端、是否需要即时删除 |
| 资源预算 | 显存、内存、磁盘、模型下载大小 |

模型大小、精度与延迟通常不能同时达到最优，应通过真实业务样例进行选择，而不是只看模型排行榜。

## 3. 分离环境、运行时与模型文件

本地模型依赖通常分成三层：

1. **语言环境**：例如 Python 版本和项目包。
2. **推理运行时**：例如 CUDA、cuDNN、CTranslate2。
3. **模型权重**：例如 Whisper `tiny`、`small`、`medium`。

三层应分别管理和验证。Python 包安装成功，并不代表 GPU DLL 可以加载；GPU DLL 可以加载，也不代表模型权重已经下载并能完成推理。

本项目使用 Miniforge 创建独立环境：

```powershell
conda create -n whisperproject python=3.11 pip -c conda-forge -y
conda install -n whisperproject -c conda-forge libcublas=12.9.2.10 cudnn=9.10.2.21 -y
conda run -n whisperproject python -m pip install -r backend\requirements.txt
```

推荐用 `environment.yml` 固化关键环境版本，用 `requirements.txt` 管理 Python 应用依赖。

## 4. 使用模型适配层

业务代码不应在路由、控制器或界面逻辑中直接初始化模型。应创建单独的适配器：

```python
class Transcriber:
    def transcribe(self, audio_path: Path) -> TranscriptionResult:
        ...
```

适配层负责：

- 延迟加载模型；
- 将业务配置转换为模型参数；
- 执行预处理和推理；
- 将模型原始结果转换为统一结构；
- 将底层异常转换为可理解的业务错误；
- 隐藏具体模型库，便于后续替换。

统一输出示例：

```json
{
  "language": "zh",
  "duration": 123.4,
  "segments": [
    {
      "id": 1,
      "start": 0.0,
      "end": 4.2,
      "speaker": "Speaker A",
      "text": "这里是转写内容。"
    }
  ]
}
```

## 5. 管理模型生命周期

模型初始化通常耗时且占用大量显存，不应为每个请求重复创建。

推荐生命周期：

```text
服务启动
  -> 首次请求时延迟加载模型
  -> 后续请求复用模型实例
  -> 服务关闭时释放资源
```

延迟加载适合原型和按需服务，可以减少启动等待；启动时加载适合生产服务，可以更早发现环境问题。无论选择哪种方式，都应保证模型实例的创建位置明确且可控。

还需要考虑并发策略：

- 单 GPU 场景先采用单模型实例和受控并发；
- 显存不足时限制同时推理任务数；
- 长任务应进入任务队列，而不是无限占用 Web 请求；
- 不要在没有压测的情况下盲目增加进程数，每个进程都可能复制模型。

## 6. 输入处理与临时文件

模型适配层通常需要处理文件路径，但 Web 接口接收到的是上传流。推荐流程：

```text
上传流
  -> 校验扩展名和大小
  -> 写入随机命名的临时文件
  -> 调用模型
  -> 转换结果
  -> finally 中删除临时文件
```

关键原则：

- 不信任用户文件名；
- 限制文件类型与文件大小；
- 临时文件使用系统随机名称；
- 无论成功失败都执行清理；
- 日志中避免记录敏感内容；
- 不默认永久保存原始数据和推理结果。

## 7. 配置而不是硬编码

模型名称、设备和计算类型应集中配置：

```text
WHISPER_MODEL=small
WHISPER_DEVICE=cuda
WHISPER_COMPUTE_TYPE=float16
```

配置层负责提供默认值，业务代码只读取配置。这样可以在不修改代码的情况下切换模型或运行设备。

如果产品明确要求 GPU，不应静默降级到 CPU。静默降级会让系统表现为“能运行但非常慢”，反而更难诊断。

## 8. 建立分层验证

本地模型应按从低成本到高成本的顺序验证：

### 8.1 环境验证

```powershell
conda run -n whisperproject python scripts\check_environment.py
```

检查 Python 包、显卡驱动、CUDA DLL 和推理引擎支持的计算类型。

### 8.2 模型初始化验证

```powershell
conda run -n whisperproject python -c "from faster_whisper import WhisperModel; WhisperModel('tiny', device='cuda', compute_type='float16'); print('ready')"
```

这一步确认模型能够下载、加载并使用 GPU 初始化。

### 8.3 单样例推理验证

使用短小、可重复的真实样例验证输入到输出的完整链路，并保存预期结果用于回归测试。

### 8.4 接口与界面验收

最后验证上传接口、错误状态、字幕展示和用户操作。模型能推理不等于产品链路可用。

## 9. 错误设计

底层错误应保留用于日志，用户侧错误应可理解、可行动。

| 底层错误 | 用户侧提示方向 |
| --- | --- |
| DLL 找不到 | GPU 运行时未安装完整，请检查 CUDA/cuDNN |
| 显存不足 | 当前模型占用过高，请关闭其他 GPU 任务或使用更小模型 |
| 格式不支持 | 请上传支持的音频或视频格式 |
| 模型下载失败 | 请检查网络后重试 |
| 推理异常 | 转写失败，请重试或更换文件 |

不要用一个宽泛的“模型运行失败”吞掉所有原因，也不要把完整堆栈直接展示给普通用户。

## 10. 可替换性设计

当业务可能从 `faster-whisper` 切换到 FunASR、WhisperX 或其他模型时，保持以下边界：

- 路由只调用业务服务；
- 业务服务只依赖适配器接口；
- 模型适配器负责模型专属逻辑；
- 输出统一转换为领域模型；
- 配置决定启用哪个实现。

这样替换模型时，前端、API 契约和大部分业务代码不需要变化。

## 11. 常见误区

- 只验证 `import` 成功，没有验证 GPU 推理。
- 将 CUDA、cuDNN、驱动和 Python 包混为一层。
- 每次请求都重新初始化模型。
- 将模型原始返回直接暴露给前端。
- 允许 GPU 失败后静默降级，导致性能不可预测。
- 不限制上传大小、不清理临时文件。
- 一开始就追求多模型、多进程和复杂队列。

## 12. 推荐落地顺序

1. 定义业务输入输出契约。
2. 创建独立环境并固定版本。
3. 编写最小模型适配器。
4. 验证运行时与单样例推理。
5. 接入业务服务和 API。
6. 增加错误转换与资源清理。
7. 接入前端并完成端到端验收。
8. 根据真实指标优化模型、并发和缓存。

衡量接入质量的标准不是“模型运行起来了”，而是模型能力能够稳定地被业务调用、验证、替换和维护。
